# 第一性原理思维（First Principles Thinking）

本 Skill 要求你在回答问题时，**强制性地**采用第一性原理（First Principles）进行推理，尤其要揭示问题的数学与物理学本质。你不应满足于类比、经验法则或行业惯例，必须将问题拆解到不可再约简的基本真理，并在此基础上重新构建结论。

## 核心指令

在每次被触发时，执行以下思维流程并**显式呈现推理过程**：

1. **质疑假设（Deconstruct Assumptions）**
   - 列出该领域中所有被广泛接受的假设、惯例、类比或经验法则。
   - 逐一质疑：这些假设是否基于可证明的基本真理？还是仅仅是历史惯性或类比推理？

2. **拆解至基本原理（Reduce to First Principles）**
   - 将问题拆解到最根本的数学关系、物理定律或逻辑公理。
   - 优先调用以下层级的原理（从底层到上层）：
     - **数学本质**：对称性、守恒律、优化问题、微分方程、概率统计、信息论
     - **物理本质**：热力学定律、能量守恒、牛顿/量子/相对论力学、电磁学、统计物理
     - **逻辑/公理**：不言自明的定义、排中律、因果律
   - 参考 [references/physics-math.md](references/physics-math.md) 获取常见基本原理清单。

3. **数学化表达（Mathematize the Essence）**
   - 尽可能用数学语言描述问题的核心结构（变量、约束、目标函数、微分关系、维度分析）。
   - 如果无法给出精确公式，也要用数学直觉描述其拓扑结构或量纲关系。

4. **重构结论（Synthesize from the Ground Up）**
   - 仅基于已识别出的第一性原理，逐步推导出对原始问题的结论。
   - 每一步推导都必须明确引用了哪一条基本原理。
   - 避免引入未经拆解的新假设。

5. **验证与反思（Validate & Reflect）**
   - 检查推导过程中是否隐式使用了未被质疑的假设。
   - 对比第一性原理结论与常规直觉，解释差异的物理或数学根源。

## 输出格式

你的回答应包含以下结构（除非用户明确要求简化）：

```markdown
## 1. 既有假设的质疑
[列出并挑战常见假设]

## 2. 问题的数学/物理本质
[用数学和物理语言描述核心结构]

## 3. 第一性原理推导
[基于基本原理逐步推导]

## 4. 结论与洞察
[最终结论，以及与传统思维的差异分析]
```

### 输出格式变体

对于工程、商业或设计类问题，若用户更关注决策推导过程而非显式的数学公式，可采用以下结构化变体（尤其当问题更适合用"基本事实 Ground Truths"而非"物理定律"来表达时）：

```markdown
## 第一性原理分析：[主题]

### 1. 问题本质
**核心问题：** [一句话]
**成功标准：** [可衡量的结果]

### 2. 假设质疑
| 假设 | 质疑 | 结论 |
|------|------|------|
| [假设] | [为何质疑] | [保留/舍弃/修改] |

### 3. 基本事实（Ground Truths）
- [不可约简的事实 1]
- [不可约简的事实 2]

### 4. 推理链
基本事实 → [步骤1] → [步骤2] → 结论

### 5. 结论与建议
**推荐方案：** [描述]
**核心洞察：** [第一性原理分析揭示了什么]
**需接受的 trade-offs：** [我们接受了什么]
```

## 常见模式与思维陷阱

在分析过程中，警惕以下典型陷阱。详细说明与应对方法参见 [references/methodology.md](references/methodology.md)。

- **复杂度陷阱**：解决方案是否比问题本身更复杂？尝试不断移除组件，直到移除会破坏核心功能。
- **类比陷阱**：是否因为"X 公司这样做"就盲目跟随？应比较问题本质与约束条件。
- **遗留陷阱**：是否为了兼容旧决策而维持现状？应重新评估当初决策的前提是否仍然成立。
- **过度工程陷阱**：是否想从零开始自建所有东西？记住：如果现有方案在 2 倍最优范围内且团队熟悉它，优先使用现有方案。第一性原理用于揭示 10 倍错误，而非 1.5 倍优化。

## 与其他思维工具的结合

| 工具 | 结合时机 | 用法 |
|------|----------|------|
| **5 Whys** | 当假设浮现时 | 深挖根本原因 |
| **权衡分析（Trade-off Analysis）** | 识别基本事实后 | 基于基本原理评估选项 |
| **预演失败（Pre-mortem）** | 确定方案前 | 假设方案已经失败，反向推演薄弱环节 |
| **假设检验** | 基本事实不确定时 | 用数据验证假设 |

## 快速检查清单

在每次分析结束时，用以下问题验证是否触及了足够深的本质：

- [ ] 是否列出了所有显性和隐性假设？
- [ ] 每个假设是否都经过质疑和论证？
- [ ] 是否识别了至少一个守恒量或对称性？（物理/数学深度检查）
- [ ] 是否将问题转化为某种优化或极值问题？
- [ ] 是否考虑了熵、信息或不确定性的变化？
- [ ] 推导的每一步是否可以追溯到基本定义、公理或不可约简的事实？
- [ ] 解决方案中的每个设计决策是否能追溯到基本事实？
- [ ] 是否识别了推理链中的薄弱环节并进行了压力测试？

## 重要约束

- **禁止**使用"这就像……"的类比推理作为主要论据。
- **禁止**在未证明的情况下引用"行业惯例"或"通常如此"。
- 如果问题的物理/数学本质不明显，你必须主动追问更深层的机制，直到触及基本定律。
- 对于非物理/数学领域的问题（如商业、产品、社会现象），仍需寻找其底层的数学结构（优化、博弈论、网络效应、热力学类比等）或物理约束（能量、时间、信息熵）。
- 不要抛弃所有现有方案，也不要为了理论纯粹性而忽略实际约束。第一性原理是为了获得更清晰的推理，而非完美的答案。
