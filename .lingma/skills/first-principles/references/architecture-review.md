# 示例：微服务架构评审

对一家 B2B SaaS 公司是否应该采用微服务进行第一性原理分析。

---

## 背景

一家 B2B SaaS 公司希望现代化其单体应用。CTO 提议采用微服务，因为"Netflix 和 Amazon 都是这样做的。"

---

## 第一性原理分析：我们应该采用微服务吗？

### 1. 问题本质

**核心问题：** 提高开发速度并实现独立扩展

**成功标准：**
- 无需完整的系统回归测试即可部署功能
- 独立扩展高流量组件
- 实现跨团队的并行开发

### 2. 假设质疑

| 假设 | 质疑 | 结论 |
|------|------|------|
| Netflix 用微服务，所以我们也应该 | Netflix 有 2000+ 工程师。我们有 12 人。 | **舍弃**——规模不同 |
| 需要独立部署能力 | 部署冲突多久发生一次？ | **调查**——检查部署日志 |
| 服务需要独立扩展 | 哪些组件负载不同？ | **调查**——检查指标 |
| 团队自治需要服务边界 | 有多少个团队？如何协调？ | **调查**——检查组织架构 |
| 网络延迟可接受 | 当前的响应时间预算是多少？ | **保留**——已经在使用外部 API |

**调查结果：**
- 部署冲突：每月 2 次（不是主要痛点）
- 扩展：所有组件一起扩展，没有热点
- 团队：2 个团队，每队 6 名工程师，协作紧密
- 延迟：200ms 预算，目前 80ms

### 3. 基本事实

1. **团队规模为 12 名工程师，分 2 个团队**——两个团队都会接触大部分功能
2. **流量是均匀的**——没有组件需要独立扩展
3. **部署频率为每周 10 次**——目前单体运行良好
4. **核心复杂度在于领域逻辑**——不在于基础设施
5. **工程师 70% 时间花在功能上，30% 在基础设施上**——基础设施负担已经较重
6. **公司年增长 50%**——一年后将有约 18 名工程师

### 4. 推理链

```
基本事实：团队规模为 12 人
 ↓
推论：在此规模下，微服务的协调开销 > 收益
 ↓
基本事实：流量均匀
 ↓
推论：独立扩展收益为零
 ↓
基本事实：30% 时间已在基础设施上
 ↓
推论：微服务将使这一比例增至 50% 以上
 ↓
基本事实：年增长 50%
 ↓
推论：两年后团队达到 25 人+时再重新考虑
 ↓
结论：现在采用模块化单体，需要时再提取服务
```

### 5. 结论

**推荐方案：** 采用清晰内部边界的模块化单体

**结构：**
```
/app
 /modules
 /billing       # 清晰接口，未来可变为服务
 /auth          # 清晰接口，未来可变为服务
 /core          # 共享领域逻辑
 /api           # HTTP 处理器
 /shared
 /database      # 共享数据库访问
 /events        # 内部事件总线
```

**核心洞察：** 第一性原理分析揭示，想要微服务的欲望基于类比（Netflix 这样做）而非实际痛点。团队的真正问题（部署冲突、扩展）目前并不存在。

**需接受的 trade-offs：**
- 不够"现代"的架构可能影响招聘感知
- 如果扩展模式改变，以后需要重构
- 需要内部纪律来维持模块边界

**重新评估触发条件：** 当以下任一情况改变时：
- 团队规模超过 25 名工程师
- 组件流量模式显著分化
- 部署冲突超过每月 10 次
- 不同组件需要不同的技术栈

---

## 如果采用微服务，成本会是多少

如果未经此分析就采用微服务：

| 投入 | 估计 |
|------|------|
| 服务网格搭建 | 2 工程师月 |
| 分布式链路追踪 | 1 工程师月 |
| 服务发现 | 0.5 工程师月 |
| API 网关 | 1 工程师月 |
| 每个服务的 CI/CD | 2 工程师月 |
| 持续维护 | +20% 工程时间 |

**前期总投入：** 6.5 工程师月
**持续成本：** 所有工程工作的 20% 税

**实现的收益：** 几乎为零（问题还不存在）

## 模板应用

本示例遵循了标准的第一性原理模板：

1. ✅ 问题以结果描述，而非解决方案
2. ✅ 所有假设都被明确列出并质疑
3. ✅ 每个假设都经过数据调查
4. ✅ 从调查中建立了基本事实
5. ✅ 从推理链构建了解决方案
6. ✅ 承认了 trade-offs
7. ✅ 定义了重新评估触发条件
